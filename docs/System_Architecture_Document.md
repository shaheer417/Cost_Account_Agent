## 1. Executive Summary

The Cost Accounting Assistant is built as a **Model Context Protocol (MCP) server** that integrates with **OpenAI's GPT-4** for intelligent cost accounting analysis. The system follows a **modular, layered architecture** with clear separation of concerns, enabling maintainability, testability, and scalability.

### Key Architectural Characteristics:

- **Architecture Style:** Layered + Microservices-ready
- **Communication Protocol:** MCP (Model Context Protocol) + REST API
- **Data Storage:** PostgreSQL with Row-Level Security (RLS)
- **AI Integration:** OpenAI API (GPT-4 / GPT-3.5-turbo)
- **Authentication:** OAuth 2.1 + PKCE
- **Deployment:** Containerized (Docker) + Cloud-native

---

## 2. System Overview

### 2.1 Purpose

The system automates cost accounting calculations, performs AI-powered variance analysis, and generates intelligent financial reports for multi-tenant organizations.

### 2.2 Scope

**In Scope:**
- Cost calculations (standard, actual, ABC, job costing)
- Variance analysis (material, labor, overhead, sales)
- Financial metrics (break-even, contribution margin, ROI, CVP)
- AI-powered insights and recommendations
- Multi-tenant data management
- Report generation (Excel, PDF, JSON)

**Out of Scope (Version 1.0):**
- General ledger accounting
- Inventory management
- Production planning
- Payroll processing
- Custom web UI (API-only)

---

## 3. Architecture Principles

### 3.1 Guiding Principles

1. **Modularity:** Each component has a single, well-defined responsibility
2. **Separation of Concerns:** Business logic separated from data access and presentation
3. **Security by Design:** Authentication, authorization, and encryption at every layer
4. **Multi-Tenancy:** Complete data isolation between tenants
5. **API-First:** RESTful API design with MCP tools
6. **Cloud-Native:** Designed for containerization and cloud deployment
7. **Scalability:** Horizontally scalable architecture
8. **Testability:** Unit, integration, and end-to-end testing support

### 3.2 Quality Attributes

| Attribute | Target | Rationale |
|-----------|--------|-----------|
| **Performance** | <500ms API response | Real-time cost analysis |
| **Availability** | 99.9% uptime | Business-critical system |
| **Scalability** | 100+ concurrent users | Multi-tenant SaaS |
| **Security** | OAuth 2.1, encryption | Financial data protection |
| **Maintainability** | 80%+ test coverage | Long-term sustainability |
| **Extensibility** | Plugin architecture | Future enhancements |

### 5.2 Component Descriptions

#### 5.2.1 MCP Server Component

**Responsibility:** Expose cost accounting functionality via MCP protocol

**Sub-components:**
- **Tools:** Executable functions (e.g., `calculate_product_cost()`)
- **Resources:** Static/dynamic data sources (e.g., cost standards)
- **Prompts:** Reusable prompt templates for AI interactions

**Technology:** FastMCP framework

#### 5.2.2 Business Logic Component

**Responsibility:** Core cost accounting calculations and analysis

**Modules:**
- **Costing Engine:** Standard, actual, ABC, job costing
- **Variance Analysis:** Material, labor, overhead, sales variances
- **Allocation Engine:** Direct and indirect cost allocation
- **Metrics Calculator:** Break-even, ROI, contribution margin, CVP

**Technology:** Python 3.12+, NumPy, Pandas

#### 5.2.3 AI Services Component

**Responsibility:** OpenAI integration for intelligent insights

**Features:**
- Natural language query parsing
- Variance explanation generation
- Cost anomaly detection
- Optimization recommendations
- Predictive forecasting

**Technology:** OpenAI API (GPT-4, GPT-3.5-turbo)

#### 5.2.4 Data Access Component

**Responsibility:** Abstract database operations

**Features:**
- Repository pattern for data access
- Query builders (SQLAlchemy)
- Connection pooling (asyncpg)
- Transaction management
- Tenant context enforcement

**Technology:** SQLAlchemy, asyncpg, Pydantic

#### 5.2.5 Security Component

**Responsibility:** Authentication, authorization, and encryption

**Features:**
- OAuth 2.1 + PKCE implementation
- JWT token generation and validation
- Role-Based Access Control (RBAC)
- Audit logging
- Data encryption

**Technology:** python-jose, passlib, cryptography

### 6.3 Multi-Tenancy Strategy

**Approach:** **Row-Level Security (RLS)** in PostgreSQL

**Implementation:**
1. Every table includes a `tenant_id` column (foreign key to `companies`)
2. PostgreSQL RLS policies enforce tenant isolation at database level
3. Application sets `app.current_tenant_id` session variable
4. All queries automatically filtered by tenant context

**Benefits:**
- Data isolation enforced by database (not just application)
- Single database instance (cost-effective)
- Centralized management
- No cross-tenant data leaks

**Security Policy Example:**
```sql
CREATE POLICY tenant_isolation ON costs
  USING (tenant_id = current_setting('app.current_tenant_id')::uuid);
```

## 11. Design Decisions

### 11.1 Key Architectural Decisions

#### Decision 1: MCP Protocol

**Decision:** Use MCP (Model Context Protocol) as primary interface

**Rationale:**
- Purpose-built for AI agent interactions
- Standardized protocol for LLM integrations
- Built-in support for tools, resources, prompts
- Growing ecosystem and client support

**Alternatives Considered:**
- Pure REST API (chosen as secondary interface)
- GraphQL (too complex for this use case)

---

#### Decision 2: PostgreSQL with Row-Level Security

**Decision:** Single PostgreSQL database with RLS for multi-tenancy

**Rationale:**
- RLS enforces data isolation at database level
- Cost-effective (single database instance)
- Simpler deployment and management
- PostgreSQL has mature RLS implementation

**Alternatives Considered:**
- Separate database per tenant (too expensive)
- Application-level filtering (security risk)
- Schema-per-tenant (management overhead)

---

#### Decision 3: Layered Architecture

**Decision:** Use layered architecture (Presentation â†’ Service â†’ Business â†’ Data)

**Rationale:**
- Clear separation of concerns
- Easier testing (mock dependencies)
- Better maintainability
- Industry standard pattern

**Alternatives Considered:**
- Hexagonal architecture (too complex for V1)
- Microservices (overkill for initial scope)

---

#### Decision 4: OpenAI API (Not Self-Hosted LLM)

**Decision:** Use OpenAI API (GPT-4 / GPT-3.5-turbo)

**Rationale:**
- Proven accuracy and reliability
- No infrastructure overhead
- Regular updates and improvements
- Cost-effective for initial scale

**Alternatives Considered:**
- Self-hosted LLM (too complex, requires GPU infrastructure)
- Other cloud AI services (OpenAI has best support)

---

#### Decision 5: OAuth 2.1 + PKCE

**Decision:** Use OAuth 2.1 with PKCE for authentication

**Rationale:**
- Industry standard for modern authentication
- PKCE adds security for public clients
- Compatible with enterprise identity providers
- Supports MFA and advanced security features

**Alternatives Considered:**
- Basic Auth (insecure)
- API Keys (no user context)
- Custom authentication (reinventing the wheel)

### 12.2 Optimization Strategies

#### Database Optimization
```sql
-- Indexing strategy
CREATE INDEX idx_costs_tenant_product ON costs(tenant_id, product_id);
CREATE INDEX idx_costs_date ON costs(cost_date DESC);
CREATE INDEX idx_variances_tenant_period ON variances(tenant_id, period);

-- Connection pooling
asyncpg pool size: min=10, max=50

-- Query optimization
- Use prepared statements
- Batch queries where possible
- Implement query result caching
```

#### AI Response Caching
```python
# Cache AI responses for identical queries
cache_key = f"ai_response:{hash(prompt)}:{tenant_id}"
ttl = 3600  # 1 hour

if cached := redis.get(cache_key):
    return cached
else:
    response = await openai_client.chat.completions.create(...)
    redis.setex(cache_key, ttl, response)
    return response
```

#### Token Optimization
- Use GPT-3.5-turbo for simple queries (10x cheaper)
- Implement prompt caching
- Prune unnecessary context from prompts
- Batch similar queries
